# TechnoRogue — Project Log

## Project Overview

**TechnoRogue** is a cyberpunk card-based roguelike inspired by Slay the Spire. It is built as a pnpm monorepo with two artifacts:

- **Web (`artifacts/techno-rogue`)** — Phaser 3 + TypeScript + Vite. Browser-based combat with a card hand fan, enemy display, and STS-style UI overlay.
- **Mobile (`artifacts/techno-rogue-mobile`)** — Expo + React Native (React 0.81.5 / Expo ~54). Native companion that mirrors the core loop with a simplified touchscreen layout.

Shared game logic lives in the mobile artifact and is implemented through a singleton `Game` class plus a `GameContext` React Context, not Zustand.

## User Preferences

- Mobile app uses **landscape orientation** (locked in `app.json`).
- Mobile routing uses **Expo Router Stack only**; the `(tabs)` scaffold was removed.
- App icon is the user-supplied **TR logo** (`attached_assets/IMG_0227_1784511296581.png`) — pink/cyan gradient on dark background — replacing the earlier AI-generated icon.
- Splash screen background color: `#0a0a0f`.
- Color palette is cyberpunk/dark: `bg` `#0a0a0f`, `bgCard` `#12122a`, `cyan` `#44ddff`, `red` `#ff4455`, `yellow` `#ffdd44`, `text` `#e0e0e0`.
- TypeScript is checked with `pnpm run typecheck` in both artifacts. No new code is declared done until typechecks pass.

## Work History

### 2026-07-21 — Combat Bug Fixes & Listener Cleanup

Goal: fix the remaining combat/card interaction blockers and land the changes on the `pre-3d` branch.

#### Web — Card Hover Jitter / Click Bug
- **Problem**: `.card` CSS used `transform: rotate(0deg) translateY(-80px) scale(1.12)` on hover, which physically moved the card away from the cursor. The cursor then left the hit area, triggering `mouseleave`, snapping the card back, and causing a hover-jitter loop that made clicking nearly impossible.
- **Fix**: Refactored `CardWidget.ts` to emit two DOM elements:
  - `.card-wrapper` — holds the arc rotation (`--card-rotation`), `pointer-events: none`, and a distant `transform-origin` to create the fan effect.
  - `.card` — the interactive visual element, `pointer-events: all`, that only scales on hover (`scale(1.1) translateY(-10px)` with `transform-origin: center bottom`). The cursor stays within the hit area.
- Updated `src/styles/cards.css` with the new wrapper/inner split, removed the arc rotation from `.card`, and added explicit `:has()`-based z-index handling so the hovered card wrapper rises above neighbors.
- `HandDisplay.ts` needed no changes; it appends `widget.element` (the wrapper) to the hand container as before.

#### Mobile — Hand Cards Not Rendering
- **Problem**: The horizontal `ScrollView` containing the hand cards was in a `flexDirection: 'row'` parent but had no `flex: 1`, so it collapsed to zero width and rendered no cards.
- **Fix**: Added `style={styles.handScrollView}` with `handScrollView: { flex: 1 }` to `combat.tsx`. Cards now appear in the horizontal scroll area.

#### Mobile — First-Turn Enemy Intent
- **Problem**: Enemies started with `currentIntent = null`; intents were only chosen inside `executeEnemyTurn()`, so the first player turn showed `?` icons. The original fix called `enemy.chooseIntent()` in `startCombat()`, but that incremented `turnCount` and caused the displayed intent to differ from the first executed intent (because `executeEnemyTurn()` called `chooseIntent()` again).
- **Fix**: In `Game.ts`:
  - `startCombat()` still primes the first intent.
  - `executeEnemyTurn()` now uses `enemy.currentIntent ?? enemy.chooseIntent()` (preserving the pre-displayed intent).
  - After executing each enemy, it calls `enemy.chooseIntent()` to set the next intent for the upcoming player turn.

#### Mobile — Landscape Orientation
- **Fix**: Changed `app.json` orientation from `"portrait"` to `"landscape"` so the combat UI fits phones.

#### Mobile — Scaffold Token Fixes
- Fixed `Colors.blue` → `Colors.cyan` in `constants/colors.ts` for `CARD_TYPE_COLORS.firewall`.
- Rewrote `hooks/useColors.ts` to return the static `Colors` object instead of a light/dark palette.
- Updated `app/+not-found.tsx` and `components/ErrorFallback.tsx` to use TechnoRogue tokens (`bg`, `text`, `cyan`) instead of the scaffold tokens (`background`, `foreground`, `primary`).
- Removed a stale `../../lib/api-client-react` reference from `tsconfig.json` that blocked typecheck.

#### Web — Combat Listener Accumulation Bug
- **Problem**: Each time `CombatScene` was created or restarted, `create()` called `setupListeners()` and `setupInput()` which added new `Game.on(...)` and keyboard listeners on top of the singleton. Old `CombatScene` closures kept firing, causing unexpected navigation to `MenuScene` when the player ended a turn or clicked UI buttons.
- **Fix**: In `src/scenes/CombatScene.ts`:
  - Stored all `Game.on(...)` callbacks in instance fields (`_onStateChanged`, `_onTurnChanged`, `_onGameOver`, `_onEnemyDamaged`, `_onPlayerHit`).
  - Added a private `_cleanup()` method that calls `Game.off(...)` for every stored listener and removes the keyboard listener.
  - Kept the `shutdown()` lifecycle hook calling `_cleanup()` for the standard Phaser path.
  - Explicitly bound `_cleanup()` to `Phaser.Scenes.Events.SHUTDOWN` and `Phaser.Scenes.Events.DESTROY` from `create()` so teardown is guaranteed even if the lifecycle hook is bypassed.
  - Replaced `addEventListener` on `#restart-btn`, `#menu-btn`, and `#end-turn-btn` with `onclick` assignments so reused DOM elements cannot accumulate handlers.
